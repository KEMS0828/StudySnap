# スレッド安全性の根本修正 — App Store審査クラッシュの解消

## 原因

Firebase・Google・AppleのSDKコールバックがバックグラウンドスレッドから呼ばれた際に、メインスレッド専用の画面データを直接変更しているため、レビュー端末（新規インストール）でクラッシュが発生。

## 修正内容

### 1. 認証サービスの全コールバックをメインスレッドに統一
- Firebaseのログイン状態リスナーの処理を `Task { @MainActor in ... }` で包む
- Googleサインインのコールバック内すべての画面データ変更を `Task { @MainActor in ... }` で包む
- Appleサインインの完了処理も同様に修正
- エラー表示・ローディング表示の変更もすべてメインスレッド保証

### 2. データ通信サービスの初期化を安全に
- Firebaseデータベースとストレージの初期化を「遅延初期化」から「即時初期化」に変更し、複数スレッドから同時アクセスされてもクラッシュしないようにする

### 3. アプリ起動時の処理を安全に
- アプリ起動時のFirebase・RevenueCat設定処理をメインスレッド保証付きに修正

### 4. サブスクリプション画面の安全性確認
- RevenueCatが未設定の場合のサブスクリプション画面表示を安全にフォールバック

## これが前回と違う理由

前回までは「初期化順序」や「ガード条件」を調整する表面的な修正でした。今回は**すべてのSDKコールバック（約10箇所）のスレッド安全性を一括修正**します。クラッシュの根本原因は「どのスレッドからデータが変更されるか」であり、タイミングではありません。
